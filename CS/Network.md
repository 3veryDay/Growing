## Q: 병목(Bottleneck) 현상에 대해 설명해주세요.
* **정의:** 전체 시스템의 성능이 특정 하나의 구성 요소로 인해 제한을 받는 현상을 의미합니다.
* **유래:** 병의 몸통보다 목 부분이 좁아 물이 나가는 속도가 제한되는 것에서 기원했습니다.
* **주요 요인:** 네트워크 대역폭, 네트워크 토폴로지, 서버 CPU/메모리 사용량, 비효율적인 네트워크 구성 등.
* **현상:** 서버 이벤트 시 트래픽이 몰려 대기 시간이 길어지거나 시스템이 느려지는 현상이 대표적입니다.

> **💡 For Me: 무신사 큐(Queue) 서버 사례**
> 최근에는 병목을 해결하기 위해 메인 서버로 직접 접속하게 하지 않고, 중간에 **대기열 서버(Queue Server)**를 배치합니다. 트래픽을 순차적으로 처리하고 진입 속도를 조절하여 메인 시스템의 다운을 방지합니다.

## Q: 네트워크 장애 발생 시 명령어창에서 무엇을 할 수 있나요?
* **ping:** 특정 IP나 도메인까지의 네트워크 연결성을 확인합니다 (ICMP 프로토콜 사용).
* **netstat:** 서비스 포트의 개방 여부, 라우팅 테이블, 현재 작동 중인 서비스 상태를 확인합니다.
* **nslookup:** DNS 관련 내용을 출력하며, 도메인 이름에 매핑된 IP 주소를 조회합니다.
* **tracert (traceroute):** 목적지까지 패킷이 거쳐가는 경로를 확인하여 어느 구간에서 병목이나 장애가 발생하는지 파악합니다.

## Q: TCP/IP 4계층 중 전송 계층에 대해 설명해주세요.
* **역할:** 송신자와 수신자를 연결하는 통신 서비스를 제공하며, Application 계층과 Internet 계층 사이의 중계 역할을 합니다. 계층 간 독립성을 보장하여 특정 계층의 변화가 다른 계층에 영향을 주지 않도록 설계되었습니다.
* **주요 프로토콜:**
    * **TCP:** 연결 지향적이며, **가상 회선 패킷 교환 방식**을 사용하여 패킷 순서를 보장하고 신뢰성을 높입니다.
    * **UDP:** 비연결형이며, **데이터그램 패킷 교환 방식**을 사용합니다. 각 패킷이 독립적인 경로를 가질 수 있어 순서가 보장되지 않으나 속도가 빠릅니다.

> **💡 For Me: 최신 스트리밍 트렌드**
> 과거엔 '신뢰성=TCP, 속도=UDP'였으나, 하드웨어의 발전으로 최근 실시간 스트리밍 서비스도 신뢰성을 위해 TCP를 많이 사용합니다. 구글, 유튜브 등은 UDP의 속도와 TCP의 신뢰성을 합친 **QUIC** 프로토콜을 사용하고 있습니다. (국내 치지직 등은 여전히 TCP 기반이 많음)

## Q: TCP의 연결 및 해제 과정(Handshake)을 설명해주세요.
* **연결 (3-Way Handshake):**
    1.  Client → Server: `SYN` (연결 요청)
    2.  Server → Client: `SYN + ACK` (요청 수락 및 역방향 연결 요청)
    3.  Client → Server: `ACK` (최종 수락) → 가상 회선 수립.
* **해제 (4-Way Handshake):**
    1.  Client → Server: `FIN` (연결 종료 선언)
    2.  Server → Client: `ACK` (일단 응답, 남은 데이터 전송 대기)
    3.  Server → Client: `FIN` (서버도 종료 준비 완료)
    4.  Client → Server: `ACK` (최종 종료 확인)

> **💡 면접 포인트:** 4-way 시 클라이언트가 바로 종료하지 않고 **TIME_WAIT** 상태를 가지는 이유는, 아직 도착하지 않은 패킷이 있을 수 있고(데이터 누락 방지), 마지막 ACK가 유실되었을 경우 서버가 다시 FIN을 보낼 때 대응하기 위해서입니다.

## Q: 공인 IP와 사설 IP, NAT에 대해 설명해주세요.
* **공인 IP:** ISP(통신사)에서 할당받는 전 세계 유일한 주소입니다.
* **사설 IP:** IPv4 주소 부족 문제를 해결하기 위해 특정 네트워크 안(공유기 등)에서만 사용하는 주소입니다.
* **NAT (Network Address Translation):** 사설 IP를 공인 IP로 변환하거나 그 반대로 변환하는 기술입니다. 주로 라우터에서 NAPT 방식으로 작동하여 하나의 공인 IP로 여러 기기가 인터넷을 쓸 수 있게 합니다.
* **DHCP:** IP 주소를 자동으로 할당해주는 프로토콜로, 사용 가능한 IP 범위 내에서 유동적으로 주소를 부여합니다.

## Q: HTTP의 특징과 버전별(1.0, 1.1, 2.0, 3.0) 차이는 무엇인가요?
* **주요 특징:** 요청-응답 구조, 미디어 독립성, Stateless(무상태), 지속 연결성.
* **버전별 변화:**
    * **HTTP/1.0:** 연결당 하나의 요청/응답 처리 (비효율적).
    * **HTTP/1.1:** `Keep-alive` 도입으로 연결 유지 후 여러 요청 처리 가능. 하지만 **HOL(Head Of Line) Blocking** 문제 발생.
    * **HTTP/2.0:** **Multiplexing** 도입으로 한 연결 내에서 여러 요청/응답을 병렬적으로 처리. 헤더 압축으로 성능 최적화.
    * **HTTP/3.0:** **UDP(QUIC)** 기반. TCP의 Handshake 과정이 없어 연결 수립이 매우 빠름.

## Q: "google.com" 입력 시 페이지가 브라우저에 뜨기까지의 과정
1.  **DNS 조회:** Local DNS 캐시 확인 → Root, TLD, Domain DNS 서버 순차 질의로 IP 획득.
2.  **TCP 연결:** 획득한 IP 주소와 3-way handshake를 통해 연결 수립.
3.  **보안 연결:** HTTPS인 경우 SSL/TLS handshake 수행.
4.  **HTTP 요청/응답:** GET 요청을 보내고 서버로부터 HTML, CSS, JS 수신.
5.  **브라우저 렌더링:**
    * **Parsing:** HTML/CSS를 분석하여 DOM/CSSOM 트리 생성.
    * **Render Tree:** 화면에 표시될 요소들을 결합.
    * **Layout & Paint:** 요소의 크기/위치 계산 후 픽셀로 변환하여 출력.
6.  **서버 측 인프라:** GSLB 또는 로드밸런싱을 통해 최적의 WAS로 포워딩되며, CDN을 통해 정적 컨텐츠를 빠르게 제공.

## Q: HTTPS와 SSL/TLS 암호화 과정에 대해 설명해주세요.
* **정의:** HTTP와 전송 계층 사이에 SSL/TLS 보안 계층을 추가하여 데이터를 암호화합니다.
* **주요 기능:** 암호화(도청 방지), 무결성(데이터 변조 확인), 신뢰성(CA 인증서 기반 신원 확인).
* **작동 방식:**
    * Handshake 단계에서는 **비대칭키**를 사용하여 서버 신원을 확인하고 **대칭키**를 안전하게 교환합니다.
    * 실제 데이터 전송 단계에서는 효율성을 위해 교환된 **대칭키(세션키)**를 사용하여 빠르게 암호화/복호화합니다.

## Q: HTTP의 Stateless 특성과 상태 유지 방법(Cookie, Session, JWT)
* **Stateless:** 서버가 클라이언트의 이전 요청 상태를 저장하지 않는 것. 서버 확장성(Scale-out)에 유리합니다.
* **상태 유지 방법:**
    * **Cookie:** 클라이언트 측 브라우저에 저장되는 작은 데이터.
    * **Session:** 서버 메모리/DB에 정보를 저장하고 클라이언트에게 세션 ID만 발급.
    * **JWT (Token):** 유저 정보를 암호화한 토큰 자체에 담아 전달. 별도 저장소 조회가 필요 없어 확장성에 유리.
    * **Redis (Session Clustering):** 서버가 여러 대일 때 세션 정보를 공유하기 위해 외부 In-memory 저장소인 Redis를 사용함.

## Q: 기타 핵심 네트워크 개념 (ARP, PDU, Routing, Gateway)
* **ARP:** 논리 주소(IP)를 통해 물리 주소(MAC)를 찾아주는 프로토콜. 반대는 RARP.
* **PDU (Protocol Data Unit):** 계층별 데이터 단위. (4계층: 세그먼트/데이터그램, 3계층: 패킷, 2계층: 프레임)
* **캡슐화/비캡슐화:** 상위 계층 데이터에 하위 계층 헤더를 붙여나가는 과정과 이를 분해하여 데이터를 추출하는 과정.
* **Routing Table:** 목적지 정보와 그곳으로 가기 위한 경로 리스트가 포함된 테이블.
* **Gateway:** 서로 다른 프로토콜이나 통신망을 사용하는 네트워크 간의 통로 역할.
