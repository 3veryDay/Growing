# **B-Tree와 B+Tree (Index)**

---

## 💡 핵심 개념 (Definition)
> 한 줄 요약: 거의 모든 RDBMS에서 데이터를 저장하는 방식


B-Tree(Balanced Tree)
모든 트리의 노드(중간 노드, 리프 노드)가 키와 데이터를 모두 가질 수 있는 구조입니다.

-  트리에서 데이터를 탐색할 때,ㄷ중간 노드더라도 데이터를 찾으면 바로 종료가 됩니다.
- 모든 리프 노드가 같은 레벨에 있어서 균형이 유지된다.
- 단점 : 하지만 전체 탐색이나 범위 탐색시, 모든 노드를 직접 방문해야하므로 비효율적입니다 .

B+Tree(updated balanced tree) 
데이터 저장 효율과 검색 성능을 극대화했고, mysql의 innoDB에서 사용하는 구조입니다. 

- 중간 노드에서는 실제 데이터가 없고, 자식 노드를 찾아가기 위한 인덱스 키만 경로 안내용으로저장되니다,
- 실제 데이터는 리프노드에서만 저장되니다,
- 모든 리프 노드는 더블 링크드 리스트로 저장되어서 범위 검색이나 전체 탐색에 유리합니다.


---
## 더알면 좋을 내용

1. page 단위의 저장
페이지 단위로 데이터를 저장, 읽기 때문에, 마지막 리프 노드들은 하나의 페이지에 수백개의 키로 저장된다. 그래서 트리의 높이는 최대 3-4단계로 유지됨.

2. 패아지 분할
실제 데이터가 삽입될 때, 페이지가 나뉘는 작업이 발생할 수도. 그래서 인덱스가 많을 수록 insert, update가 시간이 많이 걸림.

3. 다른 DBMS에서도 대부분 B+tree나 B-TREE를 씀.
컴퓨터의 한계인 디스크 io 작업 속도를 빨리 할 수 있기 때문에 보통 btree를 사용함.
